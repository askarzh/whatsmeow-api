# whatsmeow-api вҖ” Design

**Date:** 2026-04-30
**Status:** Approved (pending written-spec review)
**Repo:** `/home/askar/src/whatsmeow-api`
**Module:** `github.com/askar/whatsmeow-api`

## 1. Purpose

A long-running Go daemon that wraps [`whatsmeow`](https://github.com/tulir/whatsmeow) and exposes a stable JSON HTTP API plus an SSE event stream for a single WhatsApp account. The primary consumer is a separate MCP server that translates the API into MCP tools for Claude. The API is generic enough to be used by any HTTP client.

## 2. Goals

- One process owns the WhatsApp session so that pairing survives MCP-server restarts.
- Provide every capability the existing `whatsapp-mcp-go` bridge offers (send/receive/search/media/groups/realtime push) and the additional whatsmeow capabilities listed in v1 scope.
- Pluggable storage: SQLite by default; PostgreSQL (incl. Supabase) via the same interface.
- Clean isolation between whatsmeow, business logic, transport, and storage so any one of them can change without touching the others.
- Single static binary; Docker image as a deployment convenience.

## 3. Non-goals (v1)

- Multi-account / multi-tenant. One daemon = one WhatsApp account; run multiple instances if multiple accounts are needed.
- gRPC transport. REST + SSE is sufficient for the intended consumer.
- Object-storage media backend (S3, R2). Local filesystem only; an interface is in place so a backend can be added later without breaking callers.
- WhatsApp end-to-end testing in CI (not feasible without real account / phone).

### 4.1 Layer responsibilities

- **`waclient`** вҖ” the only package that imports `go.mau.fi/whatsmeow`. Owns the `*whatsmeow.Client`, registers event handlers, performs reconnect, and translates whatsmeow types into the daemon's domain types. Exposes a `WAClient` interface.
- **`service`** вҖ” pure use-case logic. Methods like `SendText`, `SendMedia`, `ListChats`, `SearchMessages`, `LoginQR`, `LoginPhonePair`, `Logout`, `Subscribe`. Depends only on `WAClient` and `Store` interfaces.
- **`transport/http`** вҖ” chi router; maps HTTP requests to service calls; JSON in/out; auth middleware; problem+json errors.
- **`transport/sse`** вҖ” SSE endpoint for incoming-message push.
- **`store`** вҖ” `Store` interface + two implementations (`sqlite`, `postgres`). Queries generated by `sqlc`; migrations driven by `golang-migrate`.
- **`media`** вҖ” on-disk storage of media payloads (the `Store` only holds metadata).

### 4.2 Why this split

Each unit can be understood without reading the others. Transports can be added (e.g., gRPC) without touching service or store. The whatsmeow upgrade blast radius is limited to the `waclient` package.

## 5. Storage

Two logical databases, each can be SQLite or Postgres:

1. **whatsmeow session store** вҖ” keys, devices, pre-keys. Use `go.mau.fi/whatsmeow/store/sqlstore` directly; supports both backends out of the box.
2. **App store** вҖ” `chats`, `messages`, `contacts`, `media`, `events_log` (for SSE resume). Our schema, our migrations.

A single DSN drives both. SQLite mode uses two files in `data_dir/`. Postgres mode uses two schemas in one database (e.g. `whatsmeow_session`, `whatsmeow_app`).

### 5.1 Tables (app store)

- `chats(jid, name, kind, last_msg_at, unread_count, archived, ...)`
- `messages(id, chat_jid, sender_jid, ts, kind, body, reply_to, edited_at, deleted_at, raw_meta_jsonb)`
- `messages_fts` вҖ” FTS5 virtual table (SQLite) / `tsvector` + GIN index (Postgres)
- `contacts(jid, push_name, full_name, business_name, ...)`
- `media(message_id, mime, size, sha256, path)`
- `events_log(seq BIGSERIAL/INTEGER PK, ts, type, payload_jsonb)` вҖ” bounded retention for SSE Last-Event-ID resume
- `kv(key, value)` вҖ” small daemon state (last sync cursor, etc.)

### 5.2 Migrations

`internal/store/migrations/` holds dialect-paired SQL files (`0001_init.sqlite.sql`, `0001_init.postgres.sql`). `golang-migrate` runs them at startup.

### 5.3 Dialect parity

`sqlc` config generates two query packages from one set of `.sql` queries plus dialect-specific overrides where necessary. Integration tests run against both dialects via `testcontainers-go` for Postgres and an in-memory SQLite file.

## 6. HTTP API (v1)

Base path `/v1`. JSON request/response. `Authorization: Bearer <token>` required if `auth.token` is configured.

### 6.1 Endpoints

| Method | Path | Purpose |
|---|---|---|
| GET    | `/v1/health`                              | Liveness (`{ok, db, wa_connected}`) |
| GET    | `/v1/status`                              | WA connection state, logged-in JID, since |
| POST   | `/v1/login/qr`                            | Start QR pairing вҶ’ returns PNG (base64) + poll URL |
| POST   | `/v1/login/phone`                         | Start pair-by-phone-code вҶ’ returns 8-char code |
| POST   | `/v1/logout`                              | Logout & wipe session |
| GET    | `/v1/chats`                               | List chats (paginated) |
| GET    | `/v1/chats/{jid}`                         | Chat details |
| GET    | `/v1/chats/{jid}/messages`                | List messages (cursor pagination) |
| GET    | `/v1/messages/search?q=...`               | Full-text search across all chats |
| POST   | `/v1/messages`                            | Send text (also reply / edit / delete via fields) |
| POST   | `/v1/messages/{id}/reactions`             | React to a message |
| POST   | `/v1/messages/{id}/read`                  | Mark message(s) as read |
| POST   | `/v1/media`                               | Upload + send media (multipart/form-data) |
| GET    | `/v1/media/{message_id}`                  | Download stored media for a message |
| GET    | `/v1/contacts`                            | List contacts |
| GET    | `/v1/contacts/search?q=...`               | Search contacts |
| GET    | `/v1/groups/{jid}/members`                | Group participants |
| POST   | `/v1/groups`                              | Create group |
| POST   | `/v1/groups/{jid}/members`                | Add or remove members |
| DELETE | `/v1/groups/{jid}/membership`             | Leave group |
| POST   | `/v1/chats/{jid}/typing`                  | Typing presence |
| GET    | `/v1/events`                              | SSE event stream (see В§7) |

### 6.2 Errors

`application/problem+json` per RFC 7807. Stable error codes (`auth.unauthorized`, `wa.not_connected`, `chat.not_found`, вҖҰ) so callers can match without parsing prose.

### 6.3 Contract

`api/openapi.yaml` is the source of truth, hand-maintained alongside handlers and verified in CI by a contract test that exercises every documented endpoint.

## 7. Realtime events (SSE)

`GET /v1/events` opens a long-lived `text/event-stream`. Event types:

- `message.received` / `message.edited` / `message.deleted`
- `reaction`
- `receipt` (delivered / read)
- `presence`
- `connection` (logged_in / disconnected / reconnecting / logged_out)

Each event carries the full payload the MCP server needs (no follow-up fetch for the typical case). 15 s heartbeat (SSE comment line). On reconnect the client may pass `Last-Event-ID`; the server resumes from `events_log.seq`. Retention default: 24 h (configurable).

## 8. Authentication

- `auth.token` empty вҶ’ no auth required (intended for `bind=127.0.0.1`).
- `auth.token` set вҶ’ all `/v1/*` endpoints require `Authorization: Bearer <token>`. Constant-time comparison.
- The daemon refuses to start with `bind != 127.0.0.1` when `auth.token` is empty (fail-safe).

## 9. Configuration

Source order (highest precedence first):
1. `WMAPI_*` environment variables
2. `./config.toml` (or `--config <path>`)
3. Built-in defaults

Keys: `server.bind`, `server.port`, `auth.token`, `storage.backend`, `storage.sqlite_path`, `storage.postgres_dsn`, `data_dir`, `log.level`, `log.format`, `events.retention_hours`, `metrics.enabled`.

Defaults: `bind=127.0.0.1`, `port=8080`, `backend=sqlite`, `sqlite_path=./data/whatsmeow-api.db`, `data_dir=./data`, `log.level=info`, `log.format=text`, `events.retention_hours=24`.

A complete annotated example lives at `config.example.toml`.

## 10. Observability

- Structured logging via `log/slog`; format selectable (`text` / `json`).
- `/v1/health` reports DB ping and WA connection status.
- Optional Prometheus `/metrics` behind `metrics.enabled` flag (not v1-blocking).

## 11. Testing strategy

- **Unit:** every package has unit tests; service layer uses fakes for `WAClient` and `Store`.
- **Store integration:** every store query test runs twice вҖ” SQLite (file) and Postgres (`testcontainers-go`) вҖ” to catch dialect drift.
- **HTTP handler:** `httptest` round-trips, no real network.
- **Contract:** `api/openapi.yaml` validated against handler implementations.
- **whatsmeow:** the `waclient` adapter is intentionally thin and is excluded from coverage gates; manual smoke tests with a real account before each release.
- **CI:** GitHub Actions runs `gofmt`, `go vet`, `staticcheck`, `go test ./... -race` with the SQLite + Postgres matrix.

## 13. Dependencies (planned)

- `go.mau.fi/whatsmeow` вҖ” WhatsApp protocol client
- `github.com/go-chi/chi/v5` вҖ” HTTP router
- `github.com/spf13/cobra` вҖ” CLI subcommands
- `github.com/spf13/viper` *or* `github.com/knadh/koanf` вҖ” config (TBD; pick at scaffold)
- `github.com/golang-migrate/migrate/v4` вҖ” migrations
- `github.com/sqlc-dev/sqlc` вҖ” query generation (build-time)
- `modernc.org/sqlite` вҖ” pure-Go SQLite driver (no CGo)
- `github.com/jackc/pgx/v5` вҖ” Postgres driver
- `github.com/mdp/qrterminal/v3` вҖ” render QR in CLI for `login` subcommand
- `github.com/testcontainers/testcontainers-go` вҖ” Postgres in tests

(All choices revisitable during implementation; this list is the starting hypothesis.)
